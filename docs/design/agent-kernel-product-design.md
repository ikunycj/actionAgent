# ActionAgent Agent 内核产品设计稿（PRD）

## 0. 文档信息
- 文档名称：Agent 内核产品设计稿
- 适用范围：`actionAgent/agent` 后续实现依据（本稿仅产品设计，不含代码）
- 版本：v1.1
- 日期：2026-03-07
- 关联文档：`actionAgent/docs/actionagent-design.md`
- 借鉴基线：`openclaw` 项目（架构与运行机制借鉴，不做实现绑定）

## 1. 目标与定位
ActionAgent 内核继续对齐“低部署成本 + 高任务完成率 + 可审计分布式协作”三目标。

1. 分钟级可用：单二进制启动后可完成首个任务闭环。
2. 稳定完成任务：优先成功率、恢复率、可诊断性。
3. 多节点协作：Controller/Worker/Edge Worker 支持任务接力。
4. 默认安全：高风险能力默认关闭，审批后可执行。
5. 协议兼容：OpenAI/Anthropic 兼容，MCP/Skills 可扩展。

## 2. 借鉴原则（来自 OpenClaw）
1. 借鉴“机制”，不绑定“具体实现语言与目录”。
2. 对用户可见能力保持简洁，对内核运行时治理保持严格。
3. 所有借鉴机制必须服务本项目 KPI，不做功能堆叠。

重点借鉴机制：
1. 分层队列与并发车道（lane-aware queue）。
2. 请求幂等去重 + 生命周期双通道收敛。
3. 会话隔离与会话维护（清理/轮转/容量预算）。
4. 模型失败分类与分级 fallback。
5. 高风险执行审批令牌（allow-once / allow-always）。
6. 配置热重载规划（hot/restart/noop）。

## 3. 产品边界

### 3.1 本期范围（MVP）
1. Task Engine：接收、排队、状态机、重试、幂等。
2. Dispatcher：本地优先、能力匹配、故障迁移。
3. Model Gateway：主模型 + fallback、预算限制、失败分级。
4. Tool Runtime：工具声明、权限分级、审批执行、审计记录。
5. Memory：会话记忆 + 长期记忆 + 生命周期维护。
6. Event Bus：统一事件、统一时序、统一观测输出。
7. WebUI：任务中心、节点中心、策略中心、审计中心。

### 3.2 非目标（本期不做）
1. 可视化编排器（拖拽流程）。
2. 团队控制台全量形态（组织/SSO/RBAC 完整版）。
3. 全渠道连接器生态。

## 4. 关键用户与问题
1. 个人开发者：希望 5 分钟内跑通、日志可见、错误可定位。
2. 进阶用户：希望长任务可靠、跨设备接力、失败可补偿。
3. 团队管理员：希望权限可控、审计可导出、风险可隔离。

## 5. 内核能力设计

### 5.1 Task Engine（任务引擎）

#### 5.1.1 任务对象
1. 基础字段：`task_id`、`run_id`、`idempotency_key`、创建时间、发起端。
2. 执行字段：输入、目标、超时、优先级、预算、可用工具策略。
3. 路由字段：当前节点、候选节点、接力策略、回执目标。
4. 安全字段：风险级别、审批状态、审批记录引用。

#### 5.1.2 状态机
1. `CREATED`
2. `QUEUED`
3. `DISPATCHING`
4. `RUNNING`
5. `WAITING_APPROVAL`
6. `RETRYING`
7. `SUSPENDED`
8. `SUCCEEDED`
9. `FAILED`
10. `CANCELLED`

新增治理要求（借鉴成熟机制）：
1. 同一 `idempotency_key` 在去重窗口内只允许一个有效终态回执。
2. 终态判定采用“生命周期事件 + 去重缓存”双通道收敛，避免假失败。
3. 支持错误缓冲窗口，防止 fallback 期间误判终态。

#### 5.1.3 车道化队列
1. 默认车道：`main`。
2. 扩展车道：`cron`、`subagent`、`session:<key>`（按会话串行）。
3. 每车道可独立并发上限，支持运行期调整。
4. 关停时拒绝新入队并保留已有任务可控收敛。

默认参数（MVP 建议值）：
1. 队列等待告警阈值：2 秒。
2. 去重缓存 TTL：5 分钟。
3. 去重缓存上限：1000。

### 5.2 Dispatcher（调度分发）
1. 调度优先级：本地可用 > 能力匹配 > 延迟/成本平衡。
2. 节点模型：Controller/Worker/Edge Worker。
3. 任务接力：节点不可用时将可迁移快照转交下一节点。
4. 超时治理：任务级 timeout + 节点调用 timeout 双层控制。

### 5.3 Model Gateway（模型网关）

#### 5.3.1 路由策略
1. 主模型优先，失败后进入 fallback 链。
2. 兼容 OpenAI/Anthropic 协议，对内统一 Adapter。
3. 预算控制：token、成本、并发配额。

#### 5.3.2 失败分类
1. `rate_limit`
2. `timeout`
3. `auth` / `auth_permanent`
4. `billing`
5. `format`
6. `model_not_found`
7. `unknown`

#### 5.3.3 认证轮转
1. 同供应商多凭据轮转，优先可用凭据。
2. 冷却窗口：指数退避（防抖）。
3. 计费故障：长窗口禁用而非短重试。
4. 会话内保持粘性，减少缓存抖动。

### 5.4 Tool Runtime（工具运行时）

#### 5.4.1 权限分级
1. `L0` 读操作。
2. `L1` 低风险写操作。
3. `L2` 高风险执行（系统命令/外部副作用），默认关闭。

#### 5.4.2 审批机制
1. 支持 `allow-once` 与 `allow-always`。
2. 审批记录绑定：请求方设备、目标节点、运行上下文。
3. 一次性审批必须消费即失效，防止重放。
4. 审批超时自动失效，进入等待或失败分支。

### 5.5 Session & Memory（会话与记忆）
1. 会话 key 统一规范化，避免跨渠道串线。
2. DM 默认安全模式可配置（单用户连续性与多用户隔离可切换）。
3. 会话维护支持：`warn/enforce` 两种模式。
4. 维护动作包括：过期清理、数量上限、归档轮转、磁盘预算。
5. Memory 支持“无向量服务退化为 FTS”模式，确保可用性。
6. 读取不存在记忆文件时返回空结果而非异常，提升稳态。

### 5.6 Event Bus（事件总线）
1. 事件域：`lifecycle`、`tool`、`assistant`、`error`、`system`。
2. 每个 `run_id` 维护严格单调 `seq`，保证回放顺序。
3. 事件必须带时间戳、关联会话、关联任务。
4. 可用于实时 UI、审计导出、告警与 SLA 统计。

## 6. 分布式角色设计

### 6.1 Controller
1. 全局调度与策略中心。
2. 节点注册、健康判定、能力索引。
3. 任务回执聚合、重试决策、接力仲裁。

### 6.2 Worker
1. 执行任务、调用模型与工具。
2. 周期上报进度、心跳与结果。
3. 支持任务中断恢复与迁移接管。

### 6.3 Edge Worker
1. 服务移动端短任务场景。
2. 不承诺常驻进程，承诺可恢复服务体验。
3. 网络抖动时优先快照并延迟回执补报。

## 7. 关键流程设计

### 7.1 临时任务（目标 30 秒）
1. 客户端提交任务 + 幂等键。
2. Task Engine 入队并立即返回 `accepted`。
3. Dispatcher 本地优先执行。
4. Event Bus 持续推送执行进度。
5. 终态收敛后回执一次性确认。

### 7.2 长任务（后台）
1. 进入长任务车道并绑定预算。
2. 定时进度快照。
3. 异常触发重试、降级或接力。
4. 完成后归档审计与可追溯摘要。

### 7.3 高风险执行
1. 触发 `WAITING_APPROVAL`。
2. 审批通过后执行，拒绝或超时则失败/取消。
3. 执行全链路写审计（谁批准、批了什么、执行结果）。

## 8. WebUI 产品要求
1. 仪表盘：TTFV、成功率、平均耗时、错误分布、节点健康。
2. 任务中心：过滤、时间线、终态原因、重试/取消。
3. 节点中心：角色、能力、在线状态、最后心跳。
4. 策略中心：模型路由、预算、工具权限、审批策略。
5. 审计中心：按任务/节点/用户/时间检索导出。

## 9. 安全与治理
1. 高危工具默认关闭，显式启用。
2. 密钥使用系统安全存储（Keychain/Keystore/Credential Manager）。
3. 控制面写操作按设备与 IP 做限流。
4. 审计日志默认保留 90 天（可配置）。
5. 对外 API 稳定优先，内部可演进但须保持行为兼容。

## 10. KPI 与验收标准

### 10.1 MVP（2026 Q2-Q3）
1. TTFV <= 5 分钟。
2. 部署成功率 >= 90%。
3. 临时任务 30 秒内成功回执率 >= 95%。
4. 长任务成功率 >= 95%。

### 10.2 分布式阶段（2026 Q4）
1. 长任务成功率 >= 97%。
2. 任务接力成功率 >= 95%。
3. MTTR <= 10 分钟。

### 10.3 功能验收清单
1. 单二进制完成首个任务闭环。
2. 状态机可视、终态可解释、迁移可追溯。
3. 模型 fallback 自动生效并记录原因。
4. 高风险执行无审批不得运行。
5. 审计日志可检索、可导出、可脱敏。

## 11. 面向 `actionAgent/agent` 的实现约束（设计约束）
1. 模块边界需对齐 6 大内核组件，避免跨层耦合。
2. 模型/渠道/工具一律通过 Adapter 层接入。
3. 状态定义、事件定义、错误码集中维护。
4. 所有副作用操作必须经过 Tool Runtime 权限与审批链。
5. 可观测性为默认能力，不得后补。

## 12. 风险与待决策项
1. 节点心跳阈值与剔除策略如何平衡误杀/延迟。
2. 任务快照粒度如何平衡恢复精度与序列化成本。
3. 审计存储成本与保留策略如何分层（热/冷）。
4. 模型预算口径如何统一跨供应商计费差异。
5. iOS 控制面与执行面边界如何稳定落地。

## 13. 90 天落地节奏
1. 第 1-4 周：任务引擎、车道队列、事件总线、基础 WebUI。
2. 第 5-8 周：Controller/Worker 协作、审批流、接力与补偿。
3. 第 9-12 周：MCP/Skills 最小扩展、稳定性压测、验收与文档固化。

---
本设计稿作为 Agent 内核产品定义基线，后续扩展必须满足三条不回退原则：部署成功率、任务成功率、可审计性。
