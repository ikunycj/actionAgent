# ActionAgent 统一产品规划

## 1. 产品定位（以你的修改为准）
1. 核心定位：部署成本最低的分布式通用 Agent 平台。
2. 价值主张：客户端、网页端发起临时任务 + 电脑/云端执行长任务 + 结果回推。
3. 产品目标：让用户在不折腾复杂安装链路的前提下，快速拥有可执行、可协作、可扩展的 Agent 能力。
4. 目标用户：程序员、白领、大学生

## 2. 竞争策略（对 OpenClaw，以你的修改为准）
1. 不打“功能数量战”，打“可用率与部署体验战”，并利用开源生态和大厂差异化竞争。
2. 核心优势：
- 单二进制桌面部署，分钟级可用。
- 双平面架构（手机/电脑客户端控制面，电脑/服务器执行面）。
- 默认安全与可审计（高风险能力默认关闭，可按策略开启）。
- 分布式 Agent 协作能力（多节点任务接力）。
3. 竞争原则：
- 首发阶段优先“稳定完成任务”，不追求一次性堆满渠道与插件数量。
- 把“新手可部署率、长任务成功率、故障恢复率”作为第一竞争指标。

## 3. 兼容策略（以你的修改为准）
1. 对外 API 兼容：OpenAI API、Anthropic API。
2. 扩展层兼容：MCP、Skills。
3. 兼容落地原则：
- 对外协议稳定优先，内部实现可持续迭代。
- 采用 Adapter 适配层，避免核心与单一模型厂商或单一插件体系强耦合。

## 4. 产品形态（由前三项策略推导）
1. Core：Go Agent Kernel。
- 可单二进制部署在 Windows/Linux/macOS。
- 部署后提供独立 WebUI（参数、会话、任务、审计）。
- 可接入多个客户端控制，也可接入企业 IM 渠道（分阶段上线）。
2. Desktop / Mobile Client：
- Windows/macOS/Linux 桌面客户端：下载 Core、控制本机或远端 Core、查看状态与日志。
- iOS/Android 移动客户端：触发、查看、审批、回执，优先做控制面。
3. Cloud Relay（可选）：
- 多 Core 之间通过云中继进行通讯与协同，支撑跨网络任务接力。
4. Team Console（后续）：
- 团队权限、策略模板、审计中心、节点编排。

## 5. 核心架构（为“低部署成本 + 分布式协作”服务）
1. 内核分层：
- Task Engine：队列、状态机、重试、幂等。
- Dispatcher：任务路由到本机/远端节点。
- Model Gateway：多模型路由、fallback、预算控制。
- Tool Runtime：权限沙箱、能力声明、执行审计。
- Memory：会话记忆、长期记忆、生命周期管理。
- Event Bus：日志、指标、追踪。
2. 节点角色：
- Controller：编排与策略中心。
- Worker：任务执行节点。
- Edge Worker：移动端短时任务执行节点。
3. 关键机制：
- 断点续跑。
- 最终一致回执。
- 跨节点任务接力（Relay）。

## 6. 核心场景包（首发聚焦高频）
1. 临时任务：手机一句话触发，30 秒内回结果。
2. 长任务：网页抓取/总结/报表，后台持续执行并通知。
3. 跨设备接力：手机触发、家中电脑执行本地敏感操作。
4. 本地自动化：浏览器/文件/脚本链路（分级权限控制）。

## 7. 平台策略（遵守系统边界）
1. Windows/Linux/macOS：主执行平台，支持后台常驻。
2. Android：可持续可用（前台服务 + 调度 + 恢复），不承诺永久常驻进程。
3. iOS：控制面优先，仅在允许后台模式执行短任务。
4. 产品承诺：保证“持续服务体验”，不承诺“移动端持续进程”。

## 8. 商业化模型
1. Free：单用户、本机节点、BYOK。
2. Pro：多节点接力、任务编排、跨端同步、优先升级。
3. Team：组织权限、审计中心、策略模板、云中继。
4. 收费核心：按“协作与编排能力”收费，不按 token 差价收费。

## 9. 路线图（2026-2029）
1. 2026 Q2-Q3（MVP）：
- 单二进制部署、OpenAI/Anthropic 兼容、任务调度、基础 Web 控制台。
- KPI：TTFV <= 5 分钟，部署成功率 >= 90%。
2. 2026 Q4（分布式阶段）：
- Controller/Worker、节点配对、任务接力、失败补偿。
- KPI：长任务成功率 >= 97%。
3. 2027（产品化阶段）：
- 可视化编排、权限策略、插件签名、团队控制台。
- KPI：团队版留存、故障恢复时长。
4. 2028（生态阶段）：
- MCP 生态、模板市场、渠道连接器标准化。
- KPI：第三方插件占比、生态活跃度。
5. 2029（行业化阶段）：
- 媒体/运营/客服等场景方案，私有化与合规增强。

## 10. 技术与治理底线
1. 高危工具默认关闭，按策略显式开启。
2. 密钥全链路安全存储（Keychain/Keystore/Credential Manager）。
3. 审计日志可导出、可追溯、可脱敏。
4. 版本策略：LTS + 快速补丁双轨。
5. 兼容策略：API 稳定优先，内部实现可迭代。

## 11. 未来适应性设计（防过时）
1. 核心与适配层解耦：模型、渠道、工具可热插拔。
2. 协议中立：支持多模型供应商与多代理协议。
3. 本地优先 + 云协同：随政策与设备能力动态切换执行面。
4. 以任务语义为中心，不绑定单一 UI 或单一模型厂商。

1. 第 1-4 周：
- 完成单二进制运行时闭环（任务、调度、日志、基础 WebUI）。
- 打通 OpenAI/Anthropic 两类 API 兼容。
2. 第 5-8 周：
- 上线 Controller/Worker 基础协作与任务接力。
- 增加失败补偿与断点续跑。
3. 第 9-12 周：
- 上线 MCP/Skills 最小扩展能力。
- 发布首批模板场景（日报摘要、网页问答、提醒通知）。
