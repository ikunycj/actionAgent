## 2. 产品定义

ActionAgent 是一个用户自托管的 Agent 产品。用户下载一个 Core 发布包，启动后即可获得：

1. Go 后端运行时
2. 与 Core 同包交付、同源访问的 WebUI 管理台
3. 面向客户端、CLI 和自动化调用的统一 API

产品不依赖官方托管云服务，用户自行掌握模型接入、配置和数据。

## 3. 产品目标

1. 把部署路径压缩到“下载 -> 启动 -> 打开即用”。
2. 让原生客户端成为日常使用入口。
3. 让 WebUI 收敛为单个 Agent 的控制、管理和调试页面。
4. 让控制台安全边界明确，不把高权限浏览器页面当作普通聊天界面。
5. 保持模型厂商、API 风格和运行参数可替换，不绑定单一上游。

## 4. 产品形态与职责边界

### 4.1 Agent Core

Agent Core 是执行面、配置事实源和交付宿主。

职责：

1. 提供任务执行、模型路由、工具调用、会话、记忆、审批、审计和观测能力。
2. 保存并应用配置。
3. 对外提供 HTTP、Bridge、事件和指标接口。
4. 在生产发布中承载并提供 WebUI 静态资源。

### 4.2 WebUI

WebUI 是随 Core 一起交付的控制台，不再是独立部署产品。

职责：

1. 围绕单个 Agent 提供配置、状态、历史、任务和诊断能力。
2. 承载首次配对、会话恢复和高危操作确认等控制面流程。
3. 提供必要的调试对话能力，但不承担主要聊天产品角色。

### 4.3 Native Client

原生客户端是主要用户入口。

职责：

1. 承载日常对话、历史浏览和主要任务操作。
2. 连接本地或远程 Core。
3. 在必要时跳转到对应 Agent 的 WebUI 管理页做高级配置。

## 5. 控制面安全模型

### 5.1 总体原则

1. WebUI 是控制面，不是普通聊天页。
2. 控制面默认本地可用，默认只服务单机、单用户场景。
3. 一旦服务暴露到非回环地址，必须启用明确的远程安全模式。
4. 高危操作不能只靠“已登录”，必须具备审批和审计。
5. 密钥可写不可回显，界面和日志都不得明文泄漏。

### 5.2 默认访问模式

产品默认安全路径为：

1. Core 监听本机回环地址。
2. 用户打开同源 WebUI。
3. WebUI 通过本地配对获取控制台会话。
4. 后续通过 Cookie 会话恢复访问。

不把“默认账号密码”作为主路径，也不把浏览器长期保存 Bearer Token 作为首选方案。

### 5.3 远程访问模式

远程暴露控制面时，必须显式选择安全模式：

1. `token`：面向 CLI、脚本或受控远程调用。
2. `trusted-proxy`：面向反向代理、SSO 或企业网关后的部署。
3. `password`：仅在后续明确接受本地身份存储复杂度时再考虑。

`none` 不是正式生产模式；非 loopback 场景不得使用无认证控制面。

### 5.4 授权与审批

1. 控制面身份必须同时具备 `agent_id` 作用域和 capability scope。
2. 高危动作至少包括：
   - 配置应用
   - 密钥写入或轮换
   - 暴露面放宽
   - 高危工具或策略开关变更
3. 上述动作必须经过二次审批，并产生完整审计记录。

详细设计见 `docs/agent/SECURITY.md`。

## 6. 首发范围

1. 用户可下载并启动一个包含 WebUI 的 Core 发布包。
2. WebUI 默认作为某个 Agent 的控制台运行。
3. WebUI 可完成该 Agent 的配置查看、修改、校验和应用。
4. WebUI 可查看该 Agent 的状态、历史、任务和诊断信息。
5. 客户端可连接同一 Core，完成日常使用主链路。
6. 控制台安全链路必须覆盖：配对或认证、会话恢复、权限边界、审批、审计。

## 7. 非目标

1. 把 WebUI 做成首发阶段的多 Agent 总控平台。
2. 把 WebUI 做成主要聊天产品。
3. 把无认证远程管理作为正式能力。
4. 在首发就做多租户、团队协作或企业 IAM。
5. 依赖用户单独部署前端服务。

## 8. 验收标准

1. 用户在没有 Go 环境的机器上，能直接启动 Core 并打开 WebUI。
2. 首次控制台访问路径默认是安全的本地路径，不需要预置默认密码。
3. 控制台中的配置、任务和历史操作能明确落在某个 Agent 上下文内。
4. 高危写操作具备权限校验、审批和审计。
5. 客户端仍是主要使用入口，WebUI 主要承担控制和诊断职责。

## 9. 版本路线

### 9.1 M1：同包交付闭环

1. Core 提供内置 WebUI 静态资源。
2. 发布流程支持构建并打包 WebUI。
3. 用户无需再独立部署前端。

### 9.2 M2：控制台安全闭环

1. 定义并实现 `local-paired`、Cookie 会话和作用域模型。
2. 建立控制面审批、审计和密钥保护规则。
3. 清理“浏览器直存 token”这类仅适合开发态的假设。

### 9.3 M3：客户端主入口闭环

1. 客户端完成对话、历史和任务主链路。
2. 客户端补齐与控制台的跳转和协同。

### 9.4 M4：增强治理

1. 增强远程访问模式、代理信任和更细粒度授权。
2. 再逐步扩展多 Agent 和更复杂的治理能力。
