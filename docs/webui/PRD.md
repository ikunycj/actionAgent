## 2. 模块定位

WebUI 是随 Core 同包交付的控制台。它的定位是：

1. 单个 Agent 的配置、管理和调试页面
2. 控制面入口，而不是主要聊天入口
3. 与 Core 同源运行的高权限浏览器界面

WebUI 不再承担“先连任意 Core，再长期用浏览器完成主要产品使用”的角色。

## 3. 目标

1. 用户启动 Core 后可直接打开 WebUI。
2. WebUI 自动进入某个 Agent 的管理上下文。
3. WebUI 能完成配置查看、修改、校验和应用。
4. WebUI 能查看该 Agent 的状态、历史、任务和诊断信息。
5. WebUI 的安全路径默认适合本机、单用户控制台场景。

## 4. 功能范围

### 4.1 P0

1. 首次配对或控制面登录入口
2. 当前 Agent 的概览、状态与健康信息
3. 当前 Agent 的配置与密钥写入入口
4. 当前 Agent 的历史、任务和诊断页面
5. 高危动作确认与审批提示
6. 当前身份、当前 Agent 和 scope 的可见化展示

### 4.2 P1

1. 更强的历史筛选、对比和审计视图
2. 与客户端之间的深链接和跳转
3. 更丰富的审批队列与审计检索

## 5. 设计约束

1. 生产环境默认由 Core 同源提供 WebUI。
2. 生产主路径不要求用户手工输入 Core Base URL。
3. WebUI 所有控制动作都必须通过 Core API 完成。
4. WebUI 的主要对象是单个 Agent，而不是全局多 Agent 编排中心。
5. WebUI 默认假设自己处在高权限控制面，而不是普通内容页。

## 6. 安全要求

### 6.1 默认路径

1. 首次访问优先走 `local-paired` 配对流程。
2. 配对完成后通过控制面会话恢复访问。
3. 浏览器生产态优先使用 Cookie 会话，而不是长期保存在浏览器存储中的 Bearer Token。

### 6.2 权限展示

1. 页面必须明确显示当前 `agent_id`。
2. 页面必须明确显示当前身份和可用 scope。
3. 无权限时必须显示清晰的拒绝原因，而不是泛化成加载失败。

### 6.3 高危动作

以下动作必须在 UI 中被明确识别为高危：

1. 配置应用
2. 密钥写入与轮换
3. 高危工具策略修改
4. 网络暴露或代理信任变更

UI 必须提示这些动作会触发审批与审计，但真正的强制逻辑必须由后端完成。

### 6.4 密钥展示

1. 秘钥字段默认掩码展示。
2. 提交成功后不回填原文。
3. 历史与审计视图不显示原始密钥内容。

## 7. 非目标

1. 把 WebUI 做成主要聊天产品。
2. 把 WebUI 做成无认证远程管理面。
3. 把 WebUI 做成多 Agent 总控平台。
4. 默认依赖账号密码作为首次访问主路径。

## 8. 验收标准

1. 用户启动 Core 后可直接打开 WebUI。
2. WebUI 默认进入安全的控制面访问路径。
3. 所有控制操作都围绕当前 Agent 展开。
4. 高危动作在 UI 和后端两侧都具备明确的保护机制。
5. 用户无需单独部署前端，也无需长期手工维护浏览器里的高权限 Token。
